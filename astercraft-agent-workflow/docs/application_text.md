# 审核表单可用描述

我正在构建一个面向印刷与企业礼赠定制场景的 AI 驱动业务工作流系统，目标不是单点式聊天，而是让 Agent 参与从客户需求理解、方案构思、视觉生成、结果回传到业务数据沉淀的完整闭环。传统定制业务中，客户往往只会给出零散、模糊的描述，例如企业名称、礼赠用途、想体现的文化元素或大致风格。人工客服需要多轮追问、整理需求、转交设计师，再反复沟通修改，整个过程成本较高，也容易因为信息缺失导致设计偏差。

我设计的 Agent 系统会将客户自然语言自动拆解为结构化需求字段，包括品牌/业务信息、礼赠场景、核心视觉主体、辅助元素、色彩倾向、风格氛围、禁用元素和特殊构图要求等。系统在多轮对话中持续维护上下文状态，并根据当前信息完整度判断下一步动作：如果关键信息不足，则进行有针对性的追问；如果信息已经满足首图生成条件，则自动进入视觉方案编译流程，将客户的业务描述转化为适合图像模型理解的高质量 prompt。

在架构上，我将系统拆分为 API 接入层、Agent 决策层、结构化记忆层、Prompt 编译层、图像生成层、任务队列层、渠道回传层和业务统计层。前端渠道通过飞鸽/抖店接入真实客户消息，后端由 BrainService 负责理解意图、维护会话状态、判断是否触发生成，并将生成任务写入 outbox 队列。图像生成完成后，Worker 会自动将效果图和说明文本回传到客户窗口。

业务统计层会记录每一次客户咨询、字段收集进度、生成触发次数、图片生成成功率、人工接管次数、重新定制次数、客户修改意见和后续报价转化情况。通过这些数据，可以分析哪些产品方向更受客户关注、哪些字段最容易缺失、哪些风格或元素更容易产生有效方案，也可以反向优化 Agent 的追问策略、Prompt 编译规则和业务流程设计。这样系统不只是完成一次对话回复，而是可以持续沉淀真实客户需求，为后续产品设计、报价策略和销售转化提供依据。

后续我计划进一步拆分为多 Agent 协作模式，例如由需求理解 Agent 负责客户意图和信息抽取，视觉设计 Agent 负责纸雕结构和构图方案，报价 Agent 负责印刷工艺与成本估算，质检 Agent 负责检查生成图是否符合可制造性与品牌要求，运营分析 Agent 负责汇总客户偏好、转化数据和流程问题。目前该项目已完成核心链路设计和工程原型，后续需要大量文本推理、长上下文分析、图像生成和数据分析额度，用于测试真实客户对话、优化多轮 Agent 决策、提升方案生成质量，并逐步形成从客户需求到设计方案再到业务转化的数据闭环。
